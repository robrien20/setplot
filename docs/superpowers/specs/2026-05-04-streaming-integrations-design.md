# Streaming integrations: Apple Music links, free previews, playlist export

**Status:** approved (auto-mode brainstorm, 2026-05-04)
**Scope:** option B+C3 — personal-but-deployable, single-user OAuth flow
**Branch:** `streaming-integrations`

## Goal

For each ACR-recognized track in a set, give the user (a) an Apple Music link, (b) a 30-second in-browser preview from either streaming service without requiring login, and (c) a one-click "export the recognized tracks as a playlist on Spotify or Apple Music."

## Non-goals

- Multi-user identity inside SetPlot. SetPlot remains a single-user tool; relocating to a VPS is supported via env vars but no per-user accounts exist in SetPlot itself.
- In-browser full-track playback (Spotify Web Playback SDK or MusicKit JS playback). Out of scope for this spec.
- Apple Music export when the operator has not configured an Apple Developer key. The feature self-disables and the UI hides the button.

## Architecture

Three independent surfaces, deliberately decoupled so failure of one does not break the others:

### 1. Apple Music links (no network, no auth)

Pure UI change. The merged-track JSON already carries `apple` (Apple Music song ID). Render `https://music.apple.com/song/{apple_id}` next to the existing Spotify/YouTube links in:

- candidate cards in the "now-playing" panel (`renderNow` in `viewer/static/viewer.js`)
- track rows in the tracklist (`renderTracklist`)

### 2. 30-second previews (no auth)

Backend exposes a single endpoint: `GET /api/preview?service=apple&id=<apple_song_id>`. Returns `{ preview_url, artwork_url }` by hitting the iTunes Search lookup API (`https://itunes.apple.com/lookup?id=<id>`, no auth, no key, no quota relevant to single-user use). 30-second preview audio is served by Apple's `audio-ssl.itunes.apple.com` CDN as `audio/m4a` — playable directly with `<audio>`. Cache responses for 24h in a JSON file inside the set's data dir to avoid repeated lookups.

For Spotify, embed the official `<iframe src="https://open.spotify.com/embed/track/{spotify_id}">` — Spotify allows their iframe widget to play 30s previews of any track without auth.

UI: a small ▶ button next to each candidate's external links; clicking it inflates a tiny inline player (single shared instance, so playing a new preview tears down the previous one) and pauses the main set audio. Pressing again or seeking the main audio dismisses the preview.

### 3. Spotify playlist export (PKCE, single-user)

Auth model:
- Operator registers a Spotify app at developer.spotify.com → records the client_id.
- SetPlot config reads `SPOTIFY_CLIENT_ID` from env (or `~/.setplot/config.toml`).
- PKCE flow with redirect URI `http://127.0.0.1:8000/auth/spotify/callback` (registered in the app dashboard once).
- Tokens (`access_token`, `refresh_token`, `expires_at`) persisted to `~/.setplot/spotify_token.json`, mode 0600. Refresh handled server-side on demand.

Scopes: `playlist-modify-private playlist-modify-public`.

Track resolution (per recognized track):
1. If `spotify_id` already present from ACR → use it directly.
2. Else if `isrc` present → `GET /v1/search?type=track&q=isrc:<isrc>` and take the first hit.
3. Else → `GET /v1/search?type=track&q=track:"<title>" artist:"<artist>"`, take the first hit if score is plausible (artist substring match required to avoid bad collisions).
4. Otherwise → drop the track and report it as unmatched in the response.

Playlist creation:
- `POST /v1/users/me/playlists` with `{ name: "<set title> — SetPlot recognized tracks", description: "Generated by SetPlot from <date>", public: false }`
- `POST /v1/playlists/{id}/tracks` in batches of 100 URIs (Spotify's hard limit per request)
- Returns `{ playlist_url, matched: N, unmatched: [...] }`

### 4. Apple Music playlist export (MusicKit JS, optional)

Hard prerequisites (operator-side, one-time):
- Apple Developer membership ($99/yr)
- AuthKey `.p8` file, key ID, team ID
- Configured via `APPLE_MUSIC_KEY_PATH`, `APPLE_MUSIC_KEY_ID`, `APPLE_MUSIC_TEAM_ID` env vars (or `~/.setplot/config.toml`)

Auth model:
- Backend signs a developer JWT (ES256, exp ≤ 6 months) on demand and serves it at `GET /api/apple/dev-token`. The signing happens in-process — no shell out — using `cryptography`'s `ec` primitives (already pulled in transitively, but we'll add it explicitly).
- Frontend MusicKit JS instance configures with that developer token, then calls `await music.authorize()` which prompts the user to sign in with their Apple ID. The user music token is held in memory only (MusicKit JS handles its own storage in localStorage); SetPlot does not persist user tokens for Apple Music.
- All Apple Music catalog/library calls are made directly from the browser via `music.api.music(...)` — backend involvement ends at the developer JWT.

Track resolution:
1. If `apple_id` already present → use directly.
2. Else if `isrc` present → `GET /v1/catalog/{storefront}/songs?filter[isrc]=<isrc>` (storefront from `music.storefrontId` after authorize).
3. Else → `GET /v1/catalog/{storefront}/search?term=<title artist>&types=songs`, first match.
4. Otherwise → drop, report.

Playlist creation:
- `POST /v1/me/library/playlists` with `{ attributes: { name, description }, relationships: { tracks: { data: [{ id, type: "songs" }, ...] } } }`
- Returns `{ id, attributes }` from the API; we then build a `https://music.apple.com/library/playlist/<id>` URL for the success message.

### Self-disabling features

The `/api/sets/{id}` response gains a `services` field: `{ spotify: { enabled: bool }, apple: { enabled: bool } }`. The frontend reads this on boot and hides export buttons whose service isn't configured. No env? No buttons.

## Data flow (export Spotify)

```
viewer (Export → Spotify button)
  ↓ POST /api/export/spotify {set_id, name?, public?}
  ↓
FastAPI route
  ↓ load tokens, refresh if expired
  ↓ for tk in DATA.merged: resolve_track(tk) → spotify URI
  ↓ create playlist, add tracks in batches
  ↓
returns {playlist_url, matched, unmatched}
  ↑
viewer pops a toast with link + unmatched list
```

For Apple Music, the same flow runs entirely in the browser (no SetPlot backend involvement after the dev-token fetch).

## Testing

- Unit tests for track resolution: input merged-track shape with various combinations of `spotify_id` / `apple_id` / `isrc` / fuzzy fields; assert correct API call and proper "unmatched" labeling.
- Unit test for the Apple developer JWT: parse with PyJWT, assert claims and ES256 algorithm.
- Mock-based test for Spotify PKCE: state generation, token exchange, refresh.
- Integration test for `/api/preview`: stubs iTunes Search.
- Manual smoke: ingest a small set with known IDs, export to a Spotify test account, verify playlist appears.

## Risks and mitigations

- **iTunes Search rate limits**: undocumented, ~20 req/min observed historically. We cache aggressively (24h) and lookups are 1-per-track. A 50-track set exports once, so 50 calls — fine.
- **Spotify token leaking via the loopback redirect**: tokens land in the URL fragment of the callback only briefly; we redirect to a clean URL after the exchange. Token file is mode 0600.
- **Apple Music developer key on disk**: `.p8` files are sensitive (full Apple Music API access for the team). Document in README that the path should be outside any directory that gets committed; loading errors fail loud at startup.
- **Recognition collisions**: ISRC search can return multiple regional masters. We pick the first; document a "verify before sharing" expectation in the export success toast.

## Out of scope (won't do this PR)

- Per-user accounts in SetPlot.
- Spotify Web Playback SDK full-track playback.
- MusicKit JS in-browser full-track playback.
- Smart playlist features (deduping across exports, append-vs-replace mode).
- Tidal, YouTube Music, Deezer.
